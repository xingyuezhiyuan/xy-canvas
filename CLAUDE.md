## 编码原则

### 0. 后端同步规则

- `c:\Users\Administrator\Desktop\xymap\main.py` 是本项目的后端主入口文件。
- 当其他功能有修改的函数代码牵扯到 `main.py` 时，需要同步修改该文件。
- 修改 `main.py` 后，必须重启后端服务才能生效。

### 1. 编码前先思考

不要妄下结论。不要掩饰自己的困惑。要清楚地认识到各种权衡取舍。

**实施之前：**

- 请明确说明你的假设。如果有不确定之处，请及时询问。
- 如果存在多种解读方式，那就把它们都列出来——不要擅自做出选择。
- 如果有更简单的方法，那就直接说出来。在必要时，要敢于拒绝。
- 如果有不清楚的地方，就停下来。把不明白的地方说出来，然后提问。

### 2. 先求简洁性

解决该问题的最简代码。不包含任何不必要的复杂内容。

- 没有超出要求的功能。
- 一次性使用的代码不应使用任何抽象概念。
- 没有出现任何未经请求的"灵活性"或"可配置性"功能。
- 对于那些无法处理的情景，没有相应的错误处理机制。
- 如果你写了200行，但实际应该只需50行，那就重新撰写吧。
- 问问自己："资深工程师会认为这样做过于复杂吗？"如果会的，那就简化它吧。

### 3. 手术带来的改变

只触碰你必需接触的东西。只清理你自己弄乱的污渍。

**在编辑现有代码时：**

- 不要对周围的代码、注释或格式进行"改进"。
- 不要去重构那些没有问题的代码。
- 保持与现有风格的统一，即便你本想采用不同的处理方式。
- 如果你发现了一些无关的、无用的代码，请指出来即可，不必删除它们。

**当你的更改导致某些内容变得"孤立无援"时：**

- 删除那些因你的修改而不再被使用的导入项、变量和函数。
- 除非另有要求，否则不要删除那些已经不存在的、无用的代码。

**测试要求：** 每一处更改都应与用户的实际需求直接相关。

### 4. 以目标为导向的执行方式

确定成功标准。反复测试，直至确认符合标准为止。

**将任务转化为可验证的目标：**

- "添加验证机制" → "为无效输入编写测试用例，然后确保这些测试用例能够通过"
- "修复错误" → "先编写一个能够重现该错误的测试用例，然后再让测试用例通过"
- "重构X" → "在重构前后确保测试通过"

**对于多步骤任务，简要说明计划如下：**

1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]

明确的成功标准能让各项任务独立进行。而那些模糊不清的标准（比如"只要能实现功能就行"）则需要不断进行澄清和调整。
